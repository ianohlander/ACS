End-To-End Runtime Command Pattern

All player commands follow the same core loop: browser input becomes a typed runtime command, runtime-core mutates canonical session state, runtime-core emits events, the browser app records user-facing messages, and runtime-2d renders from a snapshot.

Keyboard or UI input
  -> apps/web command handler
  -> GameSession.dispatch(command)
  -> runtime-core state mutation and EngineEvent[]
  -> apps/web event log/message band/state panels
  -> runtime-2d canvas draw from RuntimeSnapshot

End-To-End Editor Mutation Pattern

Editor operations follow a parallel loop: a focused panel collects user intent, browser code builds a small input object, editor-core clones and updates the adventure package, validation reruns, and the current workspace rerenders from the updated draft.

Focused editor control
  -> apps/web/src/editor.ts reads form/grid input
  -> editor-core pure helper clones and updates AdventurePackage
  -> packages/validation checks references and content rules
  -> editor rerenders grid, summaries, validation, and project controls

Current Screens

Runtime Input To Rendering

Runtime input follows a narrow path: apps/web/src/index.ts turns browser input into a PlayerCommand, runtime-core mutates session state and emits events, then the browser updates panels and asks runtime-2d to redraw.

Use Case 1: Move Command

  1. The keydown handler in apps/web/src/index.ts maps Arrow keys and WASD to { type: "move", direction }.
  2. runCommand calls session.dispatch(...), appends returned events, and calls renderEverything.
  3. dispatch calls handleMove and uses the returned boolean to decide whether the command consumed a turn.
  4. handleMove computes the next coordinate, rejects bounds or occupied destinations, or updates state.player.
  5. If the destination is an exit, the engine changes currentMapId, teleports the player, and runs map-load and tile triggers.
  6. If movement succeeded, dispatch increments state.turn, emits turnEnded, and runs the enemy phase.
  7. The browser updates map name, player position, flags, inventory, dialogue, event log, and canvas.

Current behavior: blocked movement does not consume a turn and does not grant enemies a free action. Enemy behavior profiles may define turnInterval; the sample Shrine Wolf uses turnInterval: 3, so it only acts on every third successful player turn.

Use Case 2: Inspect Command

  1. The browser maps Q to { type: "inspect" }.
  2. dispatch calls handleInspect(undefined, events) and treats a completed inspection as a turn-consuming action.
  3. With no explicit direction, findEntityInDirection searches for the first active adjacent entity on the current map.
  4. If an adjacent entity exists, the engine emits inspectResult for that entity.
  5. If no adjacent entity exists, the engine emits an inspectResult describing the player's current coordinate and map id.
  6. Inspect does not directly move the player, grant items, set flags, or start dialogue.
  7. dispatch increments state.turn, emits turnEnded, then runs the enemy phase. Enemies still respect their configured turnInterval.

Use Case 3: Editor Tile Brush

  1. The designer selects Tiles mode and picks a tile id. The browser stores it in selectedTileId.
  2. renderBrushPreview updates the visible active brush.
  3. Each grid cell has a pointerdown handler in tile mode. Pressing a cell calls beginTileBrush(x, y).
  4. beginTileBrush sets isTileBrushActive = true, clears the duplicate-cell guard, and calls paintTileAt.
  5. paintTileAt skips if the editor is not in tile mode, if the same cell was already painted in this drag, or if the cell already has the selected tile.
  6. If the tile differs, the browser calls setTileAt(draft, currentMapId, x, y, tileId) from editor-core.
  7. setTileAt clones the package, finds the map and active layer, calculates index = y * map.width + x, writes the tile id, and returns a new draft.
  8. The browser refreshes the changed grid cell, reruns local validation, and updates project/publish controls.
  9. Dragging into another cell calls paintTileAt through pointerenter. Pointer up ends the brush.

Use Case 4: Editor Entity Placement

  1. The designer switches to Entities mode.
  2. The editor shows Move Entity, Add Definition, Place New, Instance Name, and Behavior Override.
  3. Choosing a reusable definition stores selectedEntityDefinitionId and enters new-instance placement mode.
  4. singleton definitions are blocked after one instance exists anywhere in the adventure.
  5. multiple definitions can be placed repeatedly.
  6. addEntityInstance clones the package, creates a unique id, applies optional displayName and behaviorOverride, pushes an EntityInstance, then validation and grid rendering rerun.

Entity Definition Vs Instance

An EntityDefinition is the reusable template. It answers, "What kind of thing is this?" Examples include Spy Handler, Police Officer, Defense Drone, and Oracle. The definition owns reusable behavior defaults, profile data, placement policy, visuals, category, and starting possessions.

An EntityInstance is the placed copy. It answers, "Which one is this, and where is it?" Instances carry map position plus optional displayName and behaviorOverride. That lets a designer place the same reusable definition more than once while naming the copies Red Team Spy Handler and Green Team Spy Handler, or using a guard override on one copy while another inherits the definition behavior.

Runtime behavior resolution first finds the entity definition, then checks the authored instance. If EntityInstance.behaviorOverride exists, the enemy turn system uses it. Otherwise it falls back to EntityDefinition.behavior and normalizes that value into an EntityBehaviorProfile.

Current note: instance creation and movement now support friendly instance names and simple per-instance behavior overrides. Browser UI deletion of instances and creation of brand-new entity definitions remain future work.

Milestone 27 Editor Information Architecture

Milestone 27 adds two practical information-architecture tools: a Selected Cell Inspector in Map Workspace and a Display Rename / Reskin workflow in Test & Publish.

The Selected Cell Inspector derives its answer from the current AdventurePackage draft. When a designer clicks a map cell, the browser stores the selected coordinate and reads the tile layer, placed entity instance, exit records, and trigger records for that same coordinate. The panel then summarizes tile name/passability, occupant, exit destination, and local triggers without creating a second state model.

The Display Rename / Reskin tool calls shared editor-core helpers. previewDisplayRename returns before/after matches for player-facing text fields. applyDisplayRename clones the draft and updates display fields only. Stable ids, trigger references, map ids, item ids, quest ids, and entity definition ids are preserved.

This is the safe path for future reskins: designers can rename visible concepts from the UI without scripts and without accidentally breaking structured references or save data.

Editor To Playtest

Painting a tile changes only the editor draft. To see it in play, the editor saves the draft to IndexedDB and opens the runtime with a draft query parameter. The runtime loads that same AdventurePackage, so there is no separate editor-only map format.

Validation, Publishing, Save, And Load

Validation and publishing

  1. Local edits rerun validateAdventure(draft).
  2. Validate Draft calls the API validation endpoint.
  3. Project save and publish are disabled when local validation is blocking.
  4. The API validates again before creating an immutable release.

Runtime save/load

  1. Save calls session.serializeSnapshot().
  2. persistence.saveSession stores the snapshot in IndexedDB.
  3. Load retrieves the record and calls engine.loadAdventure(activeAdventure, snapshot).
  4. The browser rerenders from restored state.

Future Publishing Modes

Publishing should support two different author goals without forcing them into the same artifact. A designer may want to share an editable construction-set project that another creator can fork, or they may want to ship a finished adventure as a standalone playable game with no editor surface.

ModePurposeContentsShould Exclude
forkableProjectLet another designer inspect, remix, and continue authoring the game.Editable adventure source, authoring metadata, custom libraries, starter-library provenance, dependency graph, license/remix metadata.Nothing required for responsible editing; source should remain visible.
standalonePlayableLet players run the finished adventure without seeing the construction tools.Validated immutable release, runtime assets, runtime shell, save/load support, distribution metadata.Editor UI, draft-only metadata, unused library objects, authoring diagnostics, private construction notes.

Recommended package

A future packages/publishing module should own export logic instead of putting it in the editor or runtime. Core functions should include createForkableProjectExport, createStandaloneRuntimeExport, collectRuntimeAssets, pruneUnusedAuthoringData, and validatePublishArtifact.

Release path

The safe flow is: mutable project draft, validation, immutable published release, then one or more export artifacts. Exporting must not mutate the draft or release. It should only transform a known-good release into a shareable package.

Distribution metadata

Every exported artifact should state artifactType, editable, sourceIncluded, allowForking, license, and runtimeOnly. That makes the file self-describing for humans, the app, and future AI tooling.

Genre Library Policy

The stocked starter libraries should be genre-complete enough to let a designer begin making a game immediately. The default library should include original, generic assets and definitions inspired by broad genres, not direct copies of protected intellectual property.

For example, the library can support superhero adventures with capes, secret identities, gadgets, power icons, city rooftops, and masked villains, but it should not ship Marvel or DC characters, logos, names, or distinctive protected designs. The same principle applies to science-fantasy, supernatural investigation, urban fantasy, modern spy, and other popular genres.

Current User Guide tutorial requirement

The User Guide tutorial should build a science-fiction adventure from scratch using the strongest currently supported trigger chain. The current tutorial concept is Relay Station Alecto: station maps, terminal/transit/core tiles, a station AI, a defense drone, a multi-stage relay restoration quest, dialogue/flag/item/teleport/tile-change/quest-stage triggers, presentation assets, diagnostics, and playtesting. The example should showcase current framework power without pretending future unsupported effects already exist.

Each genre pack should include
  • Terrain and architectural tiles
  • Interactable objects and treasure/items
  • NPC/enemy/entity definitions
  • Portraits or symbolic character icons
  • UI/item icons
  • Splash-screen templates
  • Starting music cues and sound-effect cue ids
  • Tags, categories, and example trigger-friendly object definitions

Milestone 24 Presentation, Pixel Art, And Starter Libraries

Milestone 24 adds the first implementation slice for classic ACS-style asset authoring and stocked starter libraries. It introduces adventure-level presentation settings, manifest-backed classic pixel sprites, and genre packs made from real reusable objects. Runtime rules, movement, triggers, and quest state remain independent from presentation data.

Milestone 28 Classic Presentation, Media Cues, And Sound Cues

Milestone 28 completes another slice of the presentation-is-not-rules architecture. The engine still knows nothing about pixels, audio devices, DOM overlays, or video playback. It only knows that a trigger action requested a media cue or sound cue, and it emits a typed event. The browser presentation layer decides how visible or audible that cue should become.

Data Model

AdventurePackage now includes mediaCues and soundCues. Each cue points at an AssetDefinition by assetId. Validation checks that referenced assets exist and warns if cue asset kinds do not match the expected visual or audible role.

flowchart LR
    Trigger["TriggerDefinition"] --> Action["Action: playMedia / playSound"]
    Action --> Cue["MediaCueDefinition or SoundCueDefinition"]
    Cue --> Asset["AssetDefinition"]
    Action --> Runtime["runtime-core EngineEvent"]
    Runtime --> Browser["apps/web event log / future playback"]

Runtime Flow

When the player interacts with the Oracle, runtime-core finds the matching trigger, emits triggerFired, applies playMedia and playSound, starts dialogue, sets flags, advances quest state, and returns the updated state plus event list to the browser.

interact command
  -> TriggerSystem
  -> mediaCuePlayed(cueId)
  -> soundCuePlayed(cueId)
  -> dialogueStarted + flagSet + questStageSet
  -> browser event log + renderer refresh

Editor Flow

The no-code trigger builder exposes Play Media Cue and Play Sound Cue in Then Actions. Choosing either action reveals only the relevant cue selector, preserving progressive disclosure.

Classic Size Flow

Classic size is renderer-local. It is stored in browser local storage and passed to CanvasGameRenderer. Maps, coordinates, tiles, entities, saves, releases, and trigger rules are unchanged.

Milestone 21 turns terrain from loose tile ids into reusable library data. A painted map cell still stores a tile id such as grass, water, shrub, or altar, but that id now resolves to a TileDefinition in the adventure package. This gives the engine, editor, validator, and renderer a shared object to reason about.

Runtime Move Against Blocked Terrain

When the player moves, runtime-core now reads the destination tile id from map layers and tile overrides, resolves that id through AdventurePackage.tileDefinitions, and emits movementBlocked with reason terrain when the definition is blocked. Blocked terrain does not consume a turn, so enemies do not receive a free action.

Editor Use Case: Create A Force Field Tile

  1. The designer opens Libraries, chooses Tiles, and creates Force Field.
  2. Browser code calls editor-core.createTileDefinition, which clones the draft and appends a sanitized TileDefinition.
  3. The designer sets passability to blocked, adds an interaction hint, tags it barrier and sci-fi, and assigns a temporary classic sprite.
  4. Validation reruns and the tile brush palette now includes the new tile by name.
  5. When the designer paints that tile and playtests, runtime-core blocks movement into that cell because it reads the same definition data the editor authored.

This keeps terrain behavior out of browser UI, keeps visuals out of runtime-core, and prepares the project for future visual modes where the same logical tile can be an 8-bit block, a high-resolution sprite, or a 3D object.

Runtime corehandleMoveAfter a successful move, checks whether the destination cell has an exit and transfers the player.
ValidationvalidateExitsReports broken target maps, out-of-bounds source/target coordinates, and suspicious overlaps.

Exit Authoring Sequence

Runtime Move Through An Exit

dispatch({ type: "move", direction: "south" })
  -> handleMove(direction, events)
  -> compute nextX/nextY
  -> reject bounds or occupied cells
  -> update player position and emit playerMoved
  -> find currentMap.exits at the new coordinate
  -> if found: set currentMapId, player.x, player.y
  -> emit teleported
  -> run map-load triggers for the target map
  -> run tile triggers at the arrival coordinate
  -> render the new snapshot

This means exits are not a renderer trick. The player first moves onto the source cell. Runtime-core then changes the canonical session state to the destination map and coordinate. After that, runtime-2d simply renders whatever the new snapshot says.

Map Graph Derivation

The map graph is intentionally derived, not stored. The editor builds rows by walking draft.maps.flatMap(map => map.exits) and rendering each source map to target map edge. That design avoids a second graph model that could drift out of sync with the real exit records.

Validation has the same source of truth. It can verify that every exit points to an existing map, that source and target coordinates are inside their maps, and that authors can notice risky patterns such as multiple exits on the same source cell.

Recommended Editor Information Architecture

The current editor works, but its panels are arranged in milestone order rather than authoring order. Map Scale And Region Ownership

The editor now uses a simpler ACS-like map hierarchy for designers: World for broad overview travel, Region for major areas such as districts or station decks, and Local Area for the playable scene map. A Local Area can still be a room, cave, street, forest clearing, ship bay, rooftop, office, shrine, or outdoor site.

Parent Region is organizational metadata. It tells the World Atlas where a map belongs, but it does not restrict movement. Exits can connect local areas inside one region, local areas in different regions, region maps, or any other valid source/target pair. The domain model can still accept older granular map kinds for compatibility, but the editor-facing flow should prefer the three-part hierarchy unless a future feature proves that more categories are necessary.

TypeScript Section And Module Organization Cleanup

Large TypeScript files should group related interfaces, exported functions, private helpers, sanitizers, and id builders by feature area. Common editors understand collapsible region comments such as //#region Exits and //#endregion.

export interface UpsertExitInput { ... }

Cleanup rules: keep interfaces immediately before the functions that use them; group code by feature area; avoid scattering helpers far below their operations; extract large regions into focused modules; and keep the cyclomatic complexity rule intact. This applies to all TypeScript code, including runtime-core, runtime-2d, content-schema, validation, persistence, project-api, apps/web, apps/api, and future packages. packages/editor-core/src/index.ts is only the first obvious candidate, not the boundary of the rule.

Progressive Disclosure UI Requirement

The corrective object model will add many more reusable objects, but the editor must not show every object and relationship at once. The UI should reveal panels and subpanels only when the current authoring task needs them.

  • Keep the top-level Edit Flow focused on one authoring area at a time.
  • Use searchable browsers and filter drawers instead of long dropdowns.
  • Show duplicate warnings as modal or inline temporary prompts, not permanent panels.
  • Place where-used graphs, raw ids, dependency reports, and diagnostics behind expandable details.
  • Prefer task-specific creation flows such as Create Objective From Template over all-purpose forms with dozens of visible fields.

Future Multiplayer Architecture

Multiplayer is planned as a later-phase capability. The recommended foundation is backend-authoritative sessions: browsers submit serializable actor commands, the backend validates them through portable runtime-core, advances canonical state, and broadcasts snapshots or event deltas back to clients.

  • Keep runtime-core DOM-free so solo browser play and server-hosted shared play can reuse the same command validation.
  • Use actor-capable action services so players, deterministic NPCs, AI NPCs, and future multiplayer actors all go through the same runtime rules.
  • Add actor context to triggers so the engine can tell which player or NPC activated an item, portal, tile, or rule.
  • Let persistence store authoritative snapshots and replay/audit events.
  • Treat peer-to-peer as optional later exploration, not the initial foundation, because conflict resolution, cheating, and reconnection are harder there.

Future Mobile Play Mode

Mobile support should mean play mode first. Phone-sized devices should not expose editor/create workflows. A future mobile shell should hide or block authoring tools through screen-size and capability gates while reusing the same runtime state and command model.

  • Touch controls should dispatch the same move, inspect, interact, continue-dialogue, and profile commands as keyboard play.
  • The play UI needs readable scaling, message-band support, current objective/state display, and an inventory/profile drawer.
  • Standalone playable publishing should be able to emit a direct play URL or installable play shell without editor source controls.
  • Larger tablets can be evaluated later, but phone editing is intentionally out of scope.
