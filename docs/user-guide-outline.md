# Proposed User Guide Structure

This outline is the proposed reader-first structure for the User Guide. The goal is onboarding, confidence, and excitement. It should feel like a clear invitation into the product, not a scattered feature dump.

## Section Intent

### 1. What WorldTree Is

- short product definition
- who it is for
- what makes it different
- simple explanation of Play Mode, Edit Mode, and Test & Publish

### 2. What You Can Do Today

- concise capability summary
- current playable and editable features
- what is already solid enough to try now

### 3. Quick Start

- how to start the web app and API
- where to open Play Mode
- where to open Edit Mode
- what a first-time reader should click first

### 4. Play The Sample Adventure

- how to move, inspect, interact, save, load, and reset
- what the sample adventure goal is
- how the runtime panels work
- enough to make the user feel the product before editing

### 5. Understand The Editor At A Glance

- explain the Edit Flow
- explain what each major area is for
- explain the difference between maps, libraries, logic, and publishing
- orient the reader before the long tutorial begins

### 6. Flagship Tutorial: Build A Showcase Adventure

This should be the centerpiece of the guide.

It should:

- sell the product
- feel creative and ambitious
- use as many currently supported features as possible
- still remain step-by-step and beginner-readable

The tutorial should not feel like a sterile checklist. It should feel like building a game someone would want to play.

#### Tutorial principles

- build something new from scratch
- make the scenario imaginative and genre-rich
- demonstrate map building, tiles, entities, dialogue, quests, triggers, exits, cues, art, testing, and publishing
- use step-specific screenshots for every important change
- use cropped screenshots where that improves clarity
- explain why each step matters, not only what to click
- call out the newest milestone features when they appear naturally

#### Tutorial chapter structure

1. The adventure pitch
2. Create the project identity
3. Build the world structure
4. Paint the main play spaces
5. Place important characters and objects
6. Create story logic and dialogue
7. Add quests and progression
8. Add media, sound, and presentation flavor
9. Validate and diagnose
10. Playtest and refine
11. Publish and export

### 7. Test, Publish, And Share

- draft vs project vs release
- validate draft
- create/save project
- publish release
- preview artifacts
- export artifacts

### 8. Forkable Vs Standalone Packages

- what each one is for
- when to use each
- how another person would use each

### 9. Working With AI Later

- explain that AI support is planned and review-first
- explain that the future AI flow will not bypass human approval
- keep this short and confidence-building, not deeply technical

### 10. Troubleshooting

- server not running
- API unavailable
- draft not saved
- release not loading
- validation blocks

### 11. Current Limits And What Is Coming Next

- concise honesty about MVP gaps
- point to future direction without overwhelming the reader

### 12. Short Glossary

- plain-language definitions of the most important terms
- no deep file/path details here

## Tutorial Quality Standard

The tutorial should be treated as a product demonstration, not only documentation.

That means:

- it should be the most exciting and creative walkthrough the current feature set can support
- it should showcase the strongest visual and systemic capabilities available now
- it should create confidence that the application can build real adventures
- it should be revised whenever milestone work changes the real workflow
