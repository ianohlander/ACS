ACS Project Roadmap

Adventure Construction Set Roadmap

This roadmap tracks the completed foundation work and the next planned milestones. The guiding idea is to evoke the original 1980s Adventure Construction Set while keeping rendering, editor UI, and simulation cleanly separated.

Strategic Direction

Vintage Feel

The project should capture the original ACS feeling through a classic gameplay panel, pixelated sprites, symbolic map icons, status rails, and a bottom message band.

Modern Architecture

The vintage look must remain a presentation mode over the same AdventurePackage, GameSessionState, and renderer-agnostic runtime-core.

Accepted Capability Placement

Media And Sound In Phase 1

Cut scenes, splash scenes, transition images/video, music cues, and sound effects belong in the content/asset/action model before MVP packaging where practical. They should be triggerable data objects, not hardcoded browser behavior.

AI-Agnostic Authoring Later

Adventure generation should depend on an abstract provider contract and validated object proposals. The editor should be able to use different LLMs or agents without rewriting the game engine.

AI NPCs As Optional Runtime

Dynamic NPC dialogue and behavior should be opt-in and bounded by authored goals, facts, fallback scripts, validation, cost/cooldown limits, and deterministic alternatives. Designer-authored NPC brain records should provide backstory, motives, relationships, voice, memory policy, and story constraints. NPCs that interact with the world must use explicit capability profiles so designers decide which items, triggers, portals, maps, and action categories each NPC may use.

Stacked Tile Effects

Trigger execution should support ordered action stacks on one cell: splash/media scene, message, item grant, item removal, stat boost, NPC spawn/despawn, tile change, quest update, sound cue, and portal travel. This belongs with trigger/action model hardening before release so classic ACS-style "Do-All" spaces can be authored without scripts.

Everything As Objects

Tiles, entities, items, spells, skills, traits, flags, objectives, rewards, dialogue, media cues, visual assets, starter packs, categories, and trigger/action templates should be CRUD-enabled objects with stable ids, category metadata, duplicate warnings, and validation. Free-text lists are only acceptable as temporary notes, not as reusable game references.

Player Profile Panel

Play mode should keep life, power, moves, current objective, and mode-visible state on screen, while also offering an open-at-need player profile panel for inventory, abilities, skills, traits, quest connections, active effects, and owned/linked objects.

Starter Library Storage

Default starter libraries should move out of sample adventure data into reusable domain/library package data. Individual adventures may include custom additions or overrides, and designers should be able to import/export custom library packs for reuse across projects.

Forkable Project Import

Forkable distributions should import as new editable projects, not overwrite the source release. The import flow should validate the artifact, show provenance and release metadata, preview included starter packs/custom libraries, and later support selective remix import of maps, quests, entities, items, tiles, assets, and library subsets.

Progressive Editor Disclosure

Every editor area should use the same left-to-right, top-to-bottom drilldown: choose the editing mode, choose the relevant object/class, then show only the controls needed for that task. Map tools should hide entity controls while painting tiles, hide tile controls while editing exits, and expose trigger/portal/entity details only when those tools are active.

Future Multiplayer

Multiplayer should use backend-authoritative sessions first. Browsers should submit serializable actor commands, the backend should validate and advance state through portable runtime-core, and clients should receive snapshots/events. Peer-to-peer can be explored later as an optional private mode, not the foundation.

Mobile Play Only

Mobile work should focus on playing finished adventures, not editing. Phone-sized devices should hide or block construction tools through screen-size and capability gates, while a responsive touch runtime shell reuses the same engine state and command model.

Pre-Milestone Review Trail

Before each milestone build, consult the durable planning documents: roadmap, System Reference, LLM project context, Codex chat log, User Guide, and engineering quality notes. Accepted recommendations should be mirrored there so project memory does not depend on chat alone.

Standalone Delivery Modes

Keep shipped standalone adventures as one portable static web bundle first. That bundle should support four delivery options over time: manual static hosting, direct hosted web sharing, a bundled local launcher that starts a tiny local web server automatically, and later desktop wrappers such as Electron or Tauri.

Remaining Feature Checklist

Rule For Future Milestone Starts

Before starting each future milestone, list the remaining feature items that the milestone is intended to move. Treat this checklist as the durable source for that start-of-milestone review, alongside the roadmap, System Reference, LLM context, and Codex chat log. Every future milestone completion must still run the full end-to-end quality gates, including npm run quality and npm test, and must always regenerate the human-facing PDF documents plus refresh screenshots so they are current, task-specific, and step-accurate before the milestone can be considered complete.

Milestone Testing Rule

Milestone 29A: Test Harness Foundation

Before further feature expansion, the project now pauses to establish full-stack testing. Every milestone completion must run npm test, or explicitly document any unavailable test layer before the milestone is considered complete.

  • npm run test:unit: package-level tests for runtime-core, editor-core, validation, and persistence.
  • npm run test:ui: headless Chromium smoke testing for the real browser editor UI.
  • npm run playtest:smoke: runtime acceptance testing for the sample adventure trigger chain and exit travel.
  • npm run test:coverage: coverage-capable unit test path; V8 coverage emission requires a verified runtime because Node 18 Windows currently crashes when raw V8 coverage is enabled.

Milestone 29B: Build Reliability And Starter Content Regression

The current Milestone 29 quality slice hardens the standard build path and protects the classic ACS starter library import from accidental regressions.

  • npm run build and npm run typecheck now use tools/build-workspace.mjs, which regenerates workspace package stubs and resolves a working TypeScript compiler from ACS_TSC, repo-local TypeScript, or the known-good Codex compiler path.
  • tests/unit/default-content.test.mjs compares imported classic ACS starter elements against legacy ACS/legacy_ACS_startpacks.txt, excluding Land of Aventuria-specific examples, so documented starter elements cannot silently disappear from the default library.
  • The remaining Milestone 29 work is documentation/PDF screenshot hardening, tutorial-as-acceptance-test refinement, and actor-capable runtime test planning.

Milestone 29C: Actor Action Readiness

This quality slice adds the first runtime-side preflight evaluator for future shared player/NPC actions. It does not make NPCs execute player commands yet; it proves the authored permission model can answer whether a proposed item, trigger, or exit action is allowed before any AI or deterministic NPC system mutates state.

  • packages/runtime-core/src/actor-permissions.ts resolves player/entity actor kind, reads capability profiles, reads target object usePolicy, and returns allowed/blocked reasons.
  • tests/unit/runtime-core.test.mjs now verifies player item use, enemy item denial, support NPC exit permission, and informational NPC item-use blocking.
  • The remaining future work is actor-owned inventories, actor-aware trigger execution, and shared actor command services that actually execute movement, item use, trigger activation, and portal traversal for non-player actors.

Milestone 29D: Documentation Validation

This quality slice turns guide screenshot expectations into an executable gate. It does not replace human visual review, but it catches broken links, missing PDFs, missing tutorial step screenshots, and repeated step screenshots before a milestone is accepted.

  • tools/validate-docs.mjs validates Markdown and HTML image references for the User Guide and System Reference.
  • npm run docs:validate verifies required PDF outputs and the Relay Station tutorial's Steps 1 through 19 screenshot sequence.
  • npm run quality now runs complexity, documentation validation, and typecheck together.

Milestone 29E: Tutorial Acceptance Manifest

This quality slice makes the User Guide tutorial itself part of milestone acceptance. The Relay Station Alecto walkthrough now has an explicit manifest listing each required step, concept, and screenshot so future UI or documentation changes cannot quietly hollow out the tutorial.

  • docs/tutorial-acceptance.json defines the tutorial contract for Steps 1 through 19.
  • tools/validate-docs.mjs reads the manifest and verifies each step title, required text cue, required screenshot, PDF output, and duplicate screenshot rule.
  • The manifest keeps the tutorial creative but testable: new milestone features can be added to the walkthrough and the acceptance file in the same commit.

Milestone 30A: Publishing Artifact Foundation

The first MVP packaging slice adds a pure publishing boundary so editable and play-only release artifacts can diverge cleanly before export UI or bundle generation is added.

  • @acs/publishing now defines createForkableProjectExport, createStandaloneRuntimeExport, collectRuntimeAssets, pruneUnusedAuthoringData, and validatePublishArtifact.
  • forkableProject artifacts preserve the full editable adventure package plus authoring-oriented metadata for remix/fork flows.
  • standalonePlayable artifacts strip starter-library authoring packs, retain runtime package data, and produce a runtime asset dependency manifest for later bundle pruning and export UI wiring.

Milestone 30B: Release Export Workflow

This slice connects the publishing boundary to the existing project/release pipeline. Designers can now export release-backed JSON artifacts from the editor without mutating the draft or bypassing release creation.

  • apps/api now exposes release artifact export through POST /api/releases/:id/artifacts.
  • @acs/project-api now exposes exportReleaseArtifact(...) so browser code does not handcraft export URLs.
  • apps/web/editor.html and apps/web/src/editor.ts now add release-backed forkable and standalone export actions in Test & Publish, enabled only when a release exists.

Milestone 30C: Standalone Play Bundle Foundation

This slice turns the standalone export from a runtime-only JSON shape into a release artifact that can describe a real static play bundle. The generated bundle still ships as JSON today, but it now contains the file manifest needed to become a standalone play-only web package.

  • packages/publishing now defines bundle-file and bundle-manifest records for standalonePlayable artifacts, plus validation that the bundle contains an entry file and packaged adventure payload.
  • apps/api/src/standalone-bundle.ts now assembles a static play-shell manifest with index.html, CSS, runtime modules, and bundle/adventure-package.json.
  • apps/api/src/index.ts now attaches that bundle manifest to standalone release exports.
  • apps/web/src/index.ts now supports booting the runtime from a ?package=... URL so a generated standalone bundle can launch its packaged adventure without talking to the editor flow.

Milestone 30D: Standalone ZIP Export

This slice turns the standalone export into an actual download-ready package. The release API still returns a typed standalone artifact, but the browser now packages its bundle manifest into a real ZIP so designers receive a distributable play bundle instead of a JSON description.

  • packages/publishing/src/standalone-archive.ts now creates a valid store-only ZIP archive from a standalone bundle manifest.
  • apps/web/src/editor.ts now downloads standalone exports as .zip while forkable exports continue to use the release-backed artifact path.
  • apps/web/editor.html now labels the action as Export Standalone ZIP.
  • Standalone packaging still begins from an immutable release and uses the bundle manifest introduced in 30C rather than bypassing the publishing boundary.

Milestone 30E: Standalone Package Preview

Milestone 30S: Artifact Integrity Report

This slice adds one final release-review checkpoint before distribution. Instead of only previewing the forkable package, standalone bundle, and shared handoff summary separately, designers can now generate one integrity report that checks whether those layers still agree.

  • packages/publishing/src/index.ts now exports createArtifactIntegrityReport(...), a pure publishing helper that compares the two export modes and their shared handoff metadata.
  • apps/web/editor.html and apps/web/src/editor.ts now add Preview Artifact Integrity, Export Integrity Report, and a dedicated Artifact Integrity Report card in Test & Publish.

  • Publishing unit tests and editor UI smoke coverage now verify the integrity-report builder plus the new preview/export controls.

Milestone 30T: Release Review Package

This slice adds one reviewer-facing deliverable on top of the new handoff and integrity documents. Designers can now preview and export a small ZIP package that bundles the shared release handoff summary and the artifact-integrity report together for external review.

  • packages/publishing/src/review-package.ts now creates a ZIP archive from that review-package manifest.
  • apps/web/editor.html and apps/web/src/editor.ts now add Preview Review Package, Export Review Package, and a dedicated Release Review Package card in Test & Publish.
  • Publishing unit tests and editor UI smoke coverage now verify the review-package manifest, ZIP contents, and new preview/export controls.

Milestone 30U: Review Package Manifest Symmetry

This slice makes the reviewer-facing ZIP self-describing too. Instead of only previewing the package contents in the editor or inferring them from the ZIP builder, the review package now includes its own manifest file and exposes that metadata in the preview and readiness text.

Proposed Editor Areas

Adventure Setup

Project identity, rules, start state, validation status, and package-wide settings.

World Atlas

Regions, maps, map categories, exits, portals, start locations, and map graph tools.

Map Workspace

Selected-map editing for terrain, tile layers, entity instances, exits, trigger markers, and selected-cell inspection.

Libraries

Reusable entity, item, tile, dialogue, quest, asset, visual manifest, and genre collection definitions. The Libraries flow should move parent-to-child: choose type, choose category/class, choose object, then edit all object details, including graphics where applicable. Genre collections should help start new adventures quickly while still allowing cross-genre borrowing.

Logic & Quests

Trigger creation, conditions, actions, quest stages, references, backlinks, and narrative flow tools.

Test & Publish

Validation, playtest launches, diagnostics, project drafts, releases, import/export, and packaging.

Architecture Principles

Renderer-Agnostic Engine

runtime-core owns commands, rules, state, events, triggers, and AI. Renderers consume state; they do not define gameplay rules.

Manifest-Based Assets

Content references stable asset IDs. Classic sprites, pixel-editor drawings, stocked genre libraries, splash screens, music cues, HD sprites, or 3D models can satisfy those references later without changing runtime rules.

Editor-Core First

New editor capabilities should start as pure operations in editor-core, then be wired into browser UI.

LLM-Readable Reference

Each milestone should refresh an AI-friendly context document that summarizes architecture, object model, flows, current features, known gaps, and future AI extension points in structured data.
