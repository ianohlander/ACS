ACS Project Roadmap

Adventure Construction Set Roadmap

This roadmap tracks the completed foundation work and the next planned milestones. The guiding idea is to evoke the original 1980s Adventure Construction Set while keeping rendering, editor UI, and simulation cleanly separated.

Strategic Direction

Vintage Feel

The project should capture the original ACS feeling through a classic gameplay panel, pixelated sprites, symbolic map icons, status rails, and a bottom message band.

Modern Architecture

The vintage look must remain a presentation mode over the same AdventurePackage, GameSessionState, and renderer-agnostic runtime-core. The editor should follow the same separation rule so future UX skins can sit over stable authoring state and editor-core operations instead of redefining editing behavior.

Accepted Capability Placement

Media And Sound In Phase 1

Cut scenes, splash scenes, transition images/video, music cues, and sound effects belong in the content/asset/action model before MVP packaging where practical. They should be triggerable data objects, not hardcoded browser behavior.

AI-Agnostic Authoring Later

Adventure generation should depend on an abstract provider contract and validated object proposals. The editor should be able to use different LLMs or agents without rewriting the game engine.

AI NPCs As Optional Runtime

Dynamic NPC dialogue and behavior should be opt-in and bounded by authored goals, facts, fallback scripts, validation, cost/cooldown limits, and deterministic alternatives. Designer-authored NPC brain records should provide backstory, motives, relationships, voice, memory policy, and story constraints. NPCs that interact with the world must use explicit capability profiles so designers decide which items, triggers, portals, maps, and action categories each NPC may use.

Stacked Tile Effects

Trigger execution should support ordered action stacks on one cell: splash/media scene, message, item grant, item removal, stat boost, NPC spawn/despawn, tile change, quest update, sound cue, and portal travel. This belongs with trigger/action model hardening before release so classic ACS-style "Do-All" spaces can be authored without scripts.

Everything As Objects

Tiles, entities, items, spells, skills, traits, flags, objectives, rewards, dialogue, media cues, visual assets, starter packs, categories, and trigger/action templates should be CRUD-enabled objects with stable ids, category metadata, duplicate warnings, and validation. Free-text lists are only acceptable as temporary notes, not as reusable game references.

Player Profile Panel

Play mode should keep life, power, moves, current objective, and mode-visible state on screen, while also offering an open-at-need player profile panel for inventory, abilities, skills, traits, quest connections, active effects, and owned/linked objects.

Scrollable Map Viewports

When a map is larger than the visible play window, runtime presentation should support true scrolling or camera-follow behavior. World coordinates should stay in shared runtime state, while viewport framing remains a presentation-layer responsibility so classic, HD, and later renderer families can each choose how to display large maps without changing gameplay rules.

Starter Library Storage

Default starter libraries should move out of sample adventure data into reusable domain/library package data. Individual adventures may include custom additions or overrides, and designers should be able to import/export custom library packs for reuse across projects.

Forkable Project Import

Forkable distributions should import as new editable projects, not overwrite the source release. The import flow should validate the artifact, show provenance and release metadata, preview included starter packs/custom libraries, and later support selective remix import of maps, quests, entities, items, tiles, assets, and library subsets.

Targeted Item Usage

Future item usage should support designer-authored targeting against self, other actors, and tiles through structured effect data, shared actor-action validation, and reusable runtime events rather than hardcoded weapon/heal logic or one-off trigger scripts.

Variable Sprite Scale

Higher-resolution renderer families should support variable-sized sprites, simple render layers, and optional pixel-accurate collision or missile-distance calculations without changing classic 8-bit mode or turning gameplay rules into renderer-specific hacks.

Progressive Editor Disclosure

Every editor area should use the same left-to-right, top-to-bottom drilldown: choose the editing mode, choose the relevant object/class, then show only the controls needed for that task. Map tools should hide entity controls while painting tiles, hide tile controls while editing exits, and expose trigger/portal/entity details only when those tools are active.

World Structure Graph Authoring

Add an optional authoring layer for overworld maps, region entry maps, local maps, and travel links without changing the core runtime rule that exits are the real graph edges. Regions should stay organizational, maps should stay playable spaces, and the world should remain a graph rather than a strict tree.

Future Multiplayer

Multiplayer should use backend-authoritative sessions first. Browsers should submit serializable actor commands, the backend should validate and advance state through portable runtime-core, and clients should receive snapshots/events. Peer-to-peer can be explored later as an optional private mode, not the foundation.

Mobile Play Only

Mobile work should focus on playing finished adventures, not editing. Phone-sized devices should hide or block construction tools through screen-size and capability gates, while a responsive touch runtime shell reuses the same engine state and command model.

Pre-Milestone Review Trail

Before each milestone build, consult the durable planning documents: roadmap, System Reference, LLM project context, Codex chat log, User Guide, and engineering quality notes. Accepted recommendations should be mirrored there so project memory does not depend on chat alone.

Architecture Health Maintenance

Run targeted architecture-health audits as the application grows so sink-file growth, cyclomatic complexity hotspots, SOLID regressions, and package-boundary leakage are caught early. Audit findings should be folded back into the roadmap as standing maintenance work instead of being left as one-off observations.

Standalone Delivery Modes

Keep shipped standalone adventures as one portable static web bundle first. That bundle should support four delivery options over time: manual static hosting, direct hosted web sharing, a bundled local launcher that starts a tiny local web server automatically, and later desktop wrappers such as Electron or Tauri.

UX Skinning Inventory

Track the live editor and runtime UI surface area in durable inventory documents so future skins know every current panel, input, list, overlay, grid, icon surface, and connection point they must support without changing function.

Milestone Testing Rule

Milestone 29A: Test Harness Foundation

Before further feature expansion, the project now pauses to establish full-stack testing. Every milestone completion must run npm test, or explicitly document any unavailable test layer before the milestone is considered complete.

  • npm run test:unit: package-level tests for runtime-core, editor-core, validation, and persistence.
  • npm run test:ui: headless Chromium smoke testing for the real browser editor UI.
  • npm run playtest:smoke: runtime acceptance testing for the sample adventure trigger chain and exit travel.
  • npm run test:coverage: coverage-capable unit test path; V8 coverage emission requires a verified runtime because Node 18 Windows currently crashes when raw V8 coverage is enabled.

Architecture Principles

Renderer-Agnostic Engine

runtime-core owns commands, rules, state, events, triggers, and AI. Renderers consume state; they do not define gameplay rules.

Manifest-Based Assets

Content references stable asset IDs. Classic sprites, pixel-editor drawings, stocked genre libraries, splash screens, music cues, HD sprites, or 3D models can satisfy those references later without changing runtime rules.

Editor-Core First

New editor capabilities should start as pure operations in editor-core, then be wired into browser UI.

LLM-Readable Reference

Each milestone should refresh an AI-friendly context document that summarizes architecture, object model, flows, current features, known gaps, and future AI extension points in structured data.
